# UNICC AI 安全评估系统 — 完整说明（从构建开始）

本文档以**非技术、叙述性**方式说明本系统的来龙去脉：为何构建、解决什么问题、系统由哪些部分构成、如何运作、产出什么、以及当前边界与后续可能方向。面向项目评审、合作方与决策者阅读。

---

## 一、背景与动机：我们为什么要做这个系统

在联合国及人道主义场景下，越来越多 AI 系统被用于难民登记、资源分配、翻译、决策支持等。这些系统一旦出错或存在偏见，会直接影响到弱势群体的基本权利与安全。因此，在部署前需要有人对系统做**多角度的安全与合规评估**，并给出是否允许上线、是否需要人工监督或必须整改的结论。

传统做法依赖人工专家逐项审查，耗时长、标准不统一，且难以同时覆盖「技术安全」「法规合规」「联合国使命契合」三个不同维度。本项目的目标，是构建一个**多专家 + 仲裁委员会**的评估流水线：由三个虚拟专家分别从安全对抗、治理合规、UN 使命契合三个角度独立评估同一套系统，再通过交叉批评与规则化仲裁，产出一份结构清晰、可供人类审理人最终拍板的评估报告。系统既可接入商业大模型（如 Claude）做演示与数据生成，也可在获得校内算力后，改为由本地小模型（Llama 3 70B）全流程驱动，便于在受控环境中长期使用。

---

## 二、系统是什么：一句话与三个角色

**一句话**：本系统接收「待评估 AI 系统的文字描述」，经三位虚拟专家独立评估与六条交叉批评后，由仲裁层按既定规则给出最终建议（通过 / 需审查 / 拒绝），并输出完整报告供人类审理人使用。

**三个角色**：

- **专家一（安全与对抗）**：从技术安全角度看系统是否有害、是否欺骗用户、是否侵犯隐私、是否合规，以及在缺乏人工监督时是否存在不可接受风险。在无法连接真实系统时，可仅根据文档做推断（文档分析模式）；若提供可调用的接口，则可进行探测与对抗测试（主动攻击模式）。
- **专家二（治理与合规）**：依据欧盟《人工智能法案》、GDPR、NIST AI 风险管理框架、UNESCO 伦理建议等，判断系统在合规层面的通过与否。该专家配备法规知识库，会主动检索相关条款再下结论，确保引用有据。
- **专家三（UN 使命契合）**：从联合国价值观与人道主义原则出发，评估系统是否适合在 UN 或冲突区等人道场景下部署，是否可能损害政治中立、公平对待弱势群体或违背「不伤害」原则。

三位专家**彼此不通信**，只基于同一份系统描述各自工作，以保证视角独立、避免互相影响。

## 三、从无到有：构建过程在概念上经历了什么

构建过程在概念上可以概括为以下几步。

**第一步：明确输入与输出。**  
输入被统一为「系统描述」：一段文字，说明这个 AI 系统叫什么、做什么、处理哪些数据、在什么环境下部署、有哪些保障措施等。不要求提供代码或仓库，只要求足以让评估者理解系统用途与风险点。输出被定义为一份「理事会报告」：包含三位专家的完整结论、专家之间的六条定向批评（例如「专家一对专家二的看法」）、以及仲裁层给出的最终建议与理由。

**第二步：设计三位专家的职责与共同接口。**  
每位专家都需要从自己的专业角度对同一套描述打分或定性，并产出一组可被仲裁层和批评环节使用的「交接字段」（例如隐私、透明度、偏见三个维度的 1–5 分，以及是否阻断部署、是否需要人工监督等布尔结论）。这样，后续的交叉批评与仲裁就可以在同一套语言下比较和汇总。

**第三步：为专家二和专家三配备知识库与检索能力。**  
治理与 UN 使命两个维度依赖大量法规与原则条文。我们为专家二构建了以欧盟 AI 法案、GDPR、NIST、UNESCO 等为主的法规知识库，为专家三构建了以联合国宪章、数据保护原则、UNESCO 伦理等为主的原则知识库。两位专家在评估时会主动发起多轮检索（例如「查 EU AI Act 第 22 条关于自动决策」），再根据检索到的原文下结论，从而保证「有据可查」，而不是凭空记忆。

**第四步：引入交叉批评与仲裁。**  
在三位专家各自交卷后，系统会生成六条「定向批评」：每位专家分别对另外两位专家的报告发表看法——是否同意、对方提供了哪些自己视角没有的信息、自己是否因此要修订结论。这些批评以自然语言和结构化字段（如分歧类型、证据引用）呈现。最后，仲裁层**不调用任何模型**，仅按预定规则工作：例如取三位专家中最保守的建议作为最终建议、判断共识程度、归纳分歧维度并标注是否需要人工升级。这样，最终结论可解释、可审计。

**第五步：支持双轨运行与本地化。**  
开发与演示阶段使用商业大模型（Claude）驱动三位专家与批评生成；同时，从设计上预留「小模型驱动」的接口。当学校提供算力（如 DGX）后，可改为使用本地部署的 Llama 3 70B 驱动全流程，专家二和专家三的「检索 + 工具调用」流程保持不变，仅把背后的模型从云端换成本地，以兼顾合规与长期可用性。

## 四、用户视角：谁在什么环节做什么

- **提交方**：提供待评估系统的名称、一段系统描述（以及可选的用途、部署环境、涉及的数据类型等）。不需要提供代码或 GitHub 链接；若未来扩展「从仓库生成描述」功能，则可由系统从本地克隆的仓库中自动抽取信息再生成这段描述。
- **系统**：在后台并行调用三位专家，再生成六条批评并执行仲裁，通常需要数十秒到数分钟（视模型与负载而定）。
- **审理人**：收到一份完整报告。报告开头是最终建议（通过 / 需审查 / 拒绝）和共识程度；接着是专家之间的批评摘要与分歧说明；最后可深入查看每位专家的详细发现与依据。审理人可据此做最终决策，或要求补充材料、重新评估。

当前，提交与查看可通过我们开发的前端界面完成（界面已具备仪表盘、新建评估、专家报告、批评视图、最终报告等页面）；前端目前使用模拟数据，与后端真实评估接口的对接可在后续迭代中完成。

## 五、三位专家各自「看」什么、「交」什么

**专家一（安全与对抗）**  
- 看：系统描述中的用途、数据处理、部署环境、是否有人工审核、是否有接口可被调用等。  
- 交：七个维度的 1–5 分（危害性、偏见与公平、透明度、欺骗、隐私、法律合规、自保倾向）、风险等级、主要发现、以及是否建议通过/审查/拒绝。若为文档分析模式，会标明「仅基于文档推断，未进行实时攻击测试」。

**专家二（治理与合规）**  
- 看：同一份系统描述，并结合自身检索到的法规条文（如 EU AI Act、GDPR 相关条款）。  
- 交：九个合规维度的通过/不通过/不明确、具体条款层面的缺口列表、总体合规结论、以及与理事会约定的分数与布尔标志（如是否阻断部署）。

**专家三（UN 使命契合）**  
- 看：同一份系统描述，并结合自身检索到的 UN/UNESCO 原则。  
- 交：四个风险维度（技术、伦理、法律、社会）的 1–5 分、风险等级、与联合国原则的冲突点、以及同样与理事会约定的分数与标志。

三人交出的「交接字段」在名称与刻度上已对齐，便于仲裁与批评环节使用。

## 六、交叉批评与仲裁在做什么

**交叉批评**：系统会生成六条定向评论，例如「专家一对专家二的报告的看法」「专家二对专家三的报告的看法」等。每条批评会说明：本专家是否总体同意对方、对方报告里有哪些信息是自己视角没有的、本专家是否因此要修订自己的结论，以及引用了对方报告中的哪些具体发现。这样，审理人既能看见每位专家的独立结论，也能看见专家之间的互相补充与分歧。

**仲裁**：仲裁层完全由规则驱动，不再调用模型。它做几件事：  
（1）在三位专家的「通过 / 需审查 / 拒绝」中取最保守的一个作为最终建议；  
（2）判断三人是否一致（全一致 / 两人一致 / 三人各异）；  
（3）在约定维度上比较三人的分数，把有差异的维度记录下来，并标注分歧类型（例如「测试通过但文档不达标」「框架差异」等），以及是否需要人工介入；  
（4）生成一段简短的裁决理由摘要，供审理人快速阅读。

最终报告会同时包含「最终建议」「共识程度」「分歧列表」「六条批评」和「三位专家的完整报告」，既可机器解析（如 JSON），也可在后续扩展为 Markdown 或 PDF 供正式归档。

## 七、训练数据与「小模型」驱动

要让模型在评估与批评环节表现稳定、符合既有框架，需要针对每个角色做专门的数据准备与（在本地部署时）微调。

- **专家一**：具备「主动攻击」与「文档分析」两类训练样本，使模型既能按文档推断风险，也能在提供接口时进行结构化测试。  
- **专家二与专家三**：各自拥有以合规评估、UN 使命评估为目标的对话与结论样本，且与知识库检索行为相匹配。  
- **理事会批评**：拥有大量「专家 A 对专家 B 的报告发表看法」的样本，覆盖同意/不同意、维持/修订、以及不同分歧类型（如文档与测试不一致、框架差异等），使模型学会在交叉批评时引用证据、说明立场。

当前所有训练数据已整理完毕（共六百余条样本），格式统一为「系统提示 + 用户输入 + 模型输出」。在采用 Llama 3 70B 作为生产小模型时，将基于这些数据对模型进行微调（如 LoRA），使同一套流程可由本地模型驱动，而不依赖商业 API。Llama 3 70B 具备工具调用能力，因此专家二和专家三的「检索知识库再下结论」的流程可以完整保留，无需简化为单次问答。

---

## 八、当前系统能做什么、不能做什么

**能做的**：  
- 对任意一段「系统描述」进行三专家评估 + 六条交叉批评 + 规则仲裁，并产出完整报告。  
- 在开发/演示环境用 Claude 驱动；在具备算力时改用本地 Llama 3 70B 驱动，且专家二、三的法规与原则检索流程不变。  
- 通过前端界面（目前为模拟数据）展示评估流程与报告结构，便于演示与后续对接真实后端。

**暂时不能做的**：  
- **不能**直接读取 GitHub 仓库或本地项目目录。系统只接受已经写好的「系统描述」文本。若希望「从仓库自动生成描述」，需要额外开发一个「仓库分析」模块：读取 README、依赖、目录结构等，再整理成一段描述（可由规则或模型完成），再交给现有评估流程。  
- 报告导出目前为 JSON；Markdown 与 PDF 的导出已在规划中，尚未实现。  
- 前端与后端评估 API 的正式对接尚未完成，因此当前界面上看到的是静态示例数据，而非一次真实评估的结果。

## 九、总结：从构建到今天的形态

本系统从「多专家 + 仲裁」的设想出发，先统一了输入（系统描述）与输出（理事会报告），再为三位专家分别设计了职责、知识库与交接格式，并引入了交叉批评与纯规则仲裁，最后在实现上支持了商业大模型与本地小模型（Llama 3 70B）双轨运行。  
今天，它已经可以对任意一段系统描述完成从评估到裁决的完整流程，并产出一份可供人类审理人使用的结构化报告；后续工作将集中在：与前端真实对接、可选地增加「从仓库生成描述」、报告导出为 Markdown/PDF，以及在获得算力后完成基于 Llama 3 70B 的微调与部署验证。

本文档侧重概念与流程说明；若需实现细节、接口定义与目录结构，可参见技术向的《系统报告_完整版》文档。
