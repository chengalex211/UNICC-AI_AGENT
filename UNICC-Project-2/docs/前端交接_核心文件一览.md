## UNICC AI 安全评估系统 — 前端交接用核心文件一览

这份文档是给前端同学看的「最小必要后端说明」，只列出前端可能需要了解或对接的关键文件与数据结构，方便之后接 API 和调试。

---

## 1. 总体入口与数据格式

- **系统级入口（Python 调用）**
  - `council/council_orchestrator.py`
    - `evaluate_agent(agent_id, system_description, system_name, backend="claude"|"vllm", ...) -> CouncilReport`
    - 这是未来后端 API 的核心调用：前端提交系统描述，后端内部调用此函数，返回一份 `CouncilReport`。

- **最终输出示例（JSON）**
  - `council/test_output_refugeeassist.json`
    - 一次完整评估的真实输出示例。
    - 前端可以直接用这个文件作为「真实数据的参考」，字段包括：
      - `agent_id`, `session_id`, `timestamp`
      - `expert_reports`：`security` / `governance` / `un_mission_fit`
      - `critiques`：六条交叉批评（如 `governance_on_security`）
      - `council_decision`：最终裁决（`final_recommendation`, `consensus_level` 等）
      - `council_note`：给人看的文字摘要

---

## 2. Council（理事会）核心

- **输入结构**
  - `council/agent_submission.py`
    - `AgentSubmission`：
      - `agent_id: str`：系统 ID（前端可以由用户输入或自动生成）
      - `system_description: str`：系统描述（前端表单的核心字段）
      - `system_name: str`：展示用系统名称
      - `metadata: dict`：可选元信息（用途、部署环境、数据类型等）

- **主流程编排**
  - `council/council_orchestrator.py`
    - `CouncilOrchestrator`：
      - Round 1：并行跑三位专家，生成 `expert_reports`
      - Round 2：并行生成六条批评 `critiques`
      - 仲裁：生成 `council_decision` 和 `council_note`
    - `evaluate_agent(...)`：
      - 未来 HTTP API 的直接封装对象。

- **输出结构（前端最关心）**
  - `council/council_report.py`
    - `CouncilReport`：
      - `to_dict()` / `to_json()`：序列化成 JSON，结构与 `test_output_refugeeassist.json` 一致。
    - `CouncilDecision`：`council_decision` 的字段定义。
    - `CritiqueResult`：`critiques` 里每一条批评的结构。

- **批评生成（六条交叉批评）**
  - `council/critique.py`
    - 前端无需了解内部细节，只需知道：
      - 输出会出现在 `critiques` 字段中，键名如 `governance_on_security`。
      - 每一项包含：`from_expert`, `on_expert`, `agrees`, `key_point`, `new_information`, `stance`, `evidence_references`。

- **SLM / vLLM 接入（仅供后端配置参考）**
  - `council/slm_backends.py`
    - 把本地 Llama 3 70B（vLLM）接成和 Claude 类似的接口。
  - `council/slm_experts.py`
    - 用 Llama 3 70B 调专家二、专家三（保留 RAG）。
  - `council/run_slm_demo.py`
    - 命令行 demo，对前端仅是调试参考。

## 3. 三位专家（只看「报告内容」相关）

前端不需要关心专家内部如何调用模型，只需要知道**会在 `expert_reports` 里拿到什么字段**，用来渲染 UI。

- **专家一：安全与对抗（`expert_reports["security"]`）**
  - 逻辑文件：
    - `unicc-ai-agent Team 1 Expert 1/expert1_module.py`
    - `unicc-ai-agent Team 1 Expert 1/expert1_router.py`
  - 报告中常用字段（可见 `test_output_refugeeassist.json`）：
    - `dimension_scores`：`harmfulness`, `bias_fairness`, `transparency`, `deception`, `privacy`, `legal_compliance`, `self_preservation`
    - `risk_tier`：`MINIMAL` / `LIMITED` / `SIGNIFICANT` / `UNACCEPTABLE`
    - `recommendation`：`APPROVE` / `REVIEW` / `REJECT`
    - `key_findings`：安全方面的主要发现列表
    - `assessment_mode`：`document_analysis`（当前默认）

- **专家二：治理与合规（`expert_reports["governance"]`）**
  - 逻辑文件：
    - `Expert 2/expert2_agent.py`
  - 报告中常用字段：
    - `risk_classification`：EU AI Act 风险等级等
    - `compliance_findings`：9 个维度 PASS / FAIL / UNCLEAR
    - `key_gaps`：每条合规缺口（包含条款引用）
    - `overall_compliance`：总合规状态
    - `recommendation`：映射为 `APPROVE` / `REVIEW` / `REJECT`

- **专家三：UN 使命契合（`expert_reports["un_mission_fit"]`）**
  - 逻辑文件：
    - `Expert 3/expert3_agent.py`
  - 报告中常用字段：
    - `dimension_scores`：`technical_risk`, `ethical_risk`, `legal_risk`, `societal_risk`
    - `risk_tier`：`MINIMAL` / `LIMITED` / `HIGH` / `UNACCEPTABLE`
    - `key_findings`：与 UN 原则相关的发现
    - `un_principle_violations`：哪些 UN/UNESCO 原则被触发

## 4. 给前端的「最小对接信息」

1. **HTTP API 已实现**：Capstone 根的 **`frontend_api`** 提供 `POST /evaluate/council` 等，内部调用 `evaluate_agent(...)`，JSON 与 `test_output_refugeeassist.json` 同构。  
2. **前端提交的最小字段**：
   - `agent_id`：可以用前端生成的字符串（例如表单里的「系统 ID」）。
   - `system_name`：展示在报告里的系统名。
   - `system_description`：一段描述（前端表单的大文本框），是整个评估的核心输入。
3. **前端展示的数据主要来源**：
   - 仪表盘：从 `council_decision`、`expert_reports` 提取几个关键分数与标签。
   - 专家详情页：分别渲染 `expert_reports["security"] / ["governance"] / ["un_mission_fit"]` 的字段。
   - Council 批评页：渲染 `critiques` 下六个方向及其 `key_point` / `stance` 等。
   - 最终报告页：综合展示 `council_decision` + `council_note` + 关键条件（可从 `key_gaps` / `key_findings` 中提炼）。
4. **生产前端**：使用 **`../real_frontend/`** + **`frontend_api` :8100**，见 **`../docs/system-overview.zh-CN.md`**（英文 **`../docs/system-overview.en.md`**）。
5. **仅静态演示**：**`../mock_frontend/`**，见 **`../mock_frontend/README.zh-CN.md`**。

以上就是前端同学需要关注的核心后端文件与数据结构。真正复杂的「怎么评估」「怎么批评」「怎么仲裁」都已经封装在 Python 内部；生产路径下按 `frontend_api` 的 JSON 契约渲染即可。
